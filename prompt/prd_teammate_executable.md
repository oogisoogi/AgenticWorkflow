## 실행 제약 (Constraints)

이 프로세스를 진행할 때 다음 제약을 유지하세요:

**조사 단계 제약**
- 각 Teammate는 **근거 없는 주장 금지** (추측/직관만으로는 안 됨)
- 기존 경쟁사/유사 사례 최소 3개 언급
- "불가능하다"는 결론은 구체적 이유 3가지 이상 제시
- 각 Teammate는 다른 관점의 영향을 받지 말 것 (독립성 유지)

**토론 단계 제약**
- Moderator는 중립 입장 유지 (특정 팀의 편 들지 않기)
- 한 관점이 연속 3번 이상 강조하면, 반대 관점에 명시적 발언 기회 제공
- "동의한다"만으로는 안 됨 → "왜 동의하는가" 근거 포함 필수
- 충돌이 없으면 안 됨 (4개 관점이 모두 일치하면 토론 깊이가 부족한 신호)

**수렴 단계 제약**
- Green Zone: **4개 관점 모두 동의** (3개만 동의하면 Yellow Zone)
- Yellow Zone: **명확한 트리거 조건** 정의 필수 ("언제 포함될 것인가?")
- Red Zone: **재검토 시점 명시** ("6개월 후" 또는 "~이면 검토")
- 기능은 최대 15개 이내로 집약 (산만해지면 안 됨)

## 성공 지표 (Success Points)

이 PRD 프로세스가 **성공했는지** 판단하는 기준:

### Phase 1 (조사) 성공 지표
✅ **각 Teammate별 결론이 구체적인가?**
- [ ] 기능명이 구체적인가 (예: "대시보드" ❌ → "매일 아침 10시 업데이트되는 주간 매출 대시보드" ✅)
- [ ] 근거가 명시되어 있는가 (예: "경쟁사 3개 분석 결과...", "사용자 인터뷰 5명 중 4명...")
- [ ] 우선순위가 명확한가 (Top 3이 명확히 도출됨)

✅ **4개 관점이 서로 다른 각도에서 봤는가?**
- [ ] Market만 강조하는 기능이 있는가? (시장 기회만 보는 것)
- [ ] User만 강조하는 기능이 있는가? (사용성만 보는 것)
- [ ] Tech만 조심하는 기능이 있는가? (기술 리스크)
- [ ] Business가 우려하는 기능이 있는가? (수익성 문제)
- → 모두 있으면 ✅ 다각적 분석 성공

### Phase 2 (토론) 성공 지표
✅ **충돌이 명확히 드러났는가?**
- [ ] 최소 3개 이상의 의견 불일치 포착됨
- [ ] 각 불일치마다 "왜 다른가"의 근거가 명시됨
- [ ] 단순 선호도 차이가 아니라 **논리적 충돌**임

✅ **합의 영역이 명확해졌는가?**
- [ ] 4개 관점이 **모두 동의**하는 기능이 최소 5개 이상
- [ ] 그 기능들이 핵심 고객 문제를 **정말 해결**하는가?
- [ ] Green Zone 기능들이 "이것들만으로도 MVP가 되나?"의 답이 "YES"

✅ **위험 가정이 포착되었는가?**
- [ ] Top 5 위험 가정이 명시됨
- [ ] 각 가정마다 "검증 방법"이 구체적임 (예: "사용자 인터뷰 10명", "경쟁사 벤치마킹")
- [ ] 위험 가정이 "비즈니스를 망칠 수 있는" 수준의 것들임

### Phase 3 (수렴) 성공 지표
✅ **PRD가 실행 가능한가?**
- [ ] Green Zone 기능들의 합계 개발 시간이 명시되고 **현실적**인가?
- [ ] 각 기능의 우선순위 순서가 명확한가?
- [ ] Phase 2로 미룬 기능들이 **정말 미룰 수 있는** 것들인가? (필수는 아닌가?)

✅ **모든 팀이 서명할 수 있는가?**
- [ ] Market Analyst: "이 PRD로 시장 기회를 잡을 수 있다" → 동의하는가?
- [ ] User Researcher: "이 기능들로 사용자 문제가 해결되는가?" → 동의하는가?
- [ ] Tech Architect: "6개월에 이 정도는 만들 수 있는가?" → 동의하는가?
- [ ] Business Strategist: "이 조합으로 돈을 벌 수 있는가?" → 동의하는가?
- → 모두 "동의"해야 ✅

✅ **미래에 후회하지 않을 결정인가?**
- [ ] 왜 이 기능을 택했고 왜 뺐는지가 명시되어 있는가?
- [ ] 만약 이 가정이 틀렸을 때 어떻게 할 것인가?
- [ ] 6개월 후 "왜 이걸 한 거야?"라는 물음에 답할 수 있는가?

## 최종 검수 체크리스트

PRD가 완성되었을 때 이것을 확인하세요:

```
조사 단계:
[ ] Market Researcher 결론이 구체적인가?
[ ] User Researcher의 페르소나 정의가 생생한가?
[ ] Tech Architect의 추정이 보수적인가?
[ ] Business Strategist의 수익화 논리가 타당한가?

토론 단계:
[ ] 최소 3개 이상의 의견 불일치가 기록되어 있는가?
[ ] 각 불일치에 대한 해결책 또는 우선순위가 결정되어 있는가?
[ ] Top 5 위험 가정과 검증 계획이 명시되어 있는가?

수렴 단계:
[ ] Green Zone 기능이 5-7개로 집약되어 있는가?
[ ] 각 Green Zone 기능이 "반드시 필요"한 이유가 명시되어 있는가?
[ ] Yellow Zone 기능에 "포함 조건"이 명시되어 있는가?
[ ] Red Zone 기능에 "재검토 시점"이 명시되어 있는가?

최종:
[ ] 4명의 Teammate 모두가 이 PRD에 동의할 것 같은가?
[ ] 6개월 후 "우리가 한 선택이 맞았다"고 판단할 것 같은가?
[ ] 만약 PRD를 벗어나는 요청이 들어오면, 이 문서로 거절할 수 있는가?
```

## 당신의 임무: 구조화된 토론 진행

### STEP 1: 합의 영역 파악 (Overlap Analysis)

질문: "4개 관점이 모두 강조하는 기능/우선순위는?"

아래 항목별로 분석:
- 모든 팀이 TOP 3에 넣은 기능?
- 모두가 "이건 6개월 내 불가능하다" 하는 건?
- 모두가 "이게 성공 신호다" 하는 KPI?

**형식:**
```
✅ 전체 합의 항목:
1. [기능]: Market[동의], User[동의], Tech[동의], Business[동의]
2. [기능]: ...

⚠️ 부분 합의 (3개만 동의):
1. [기능]: Market[동의], User[동의], Tech[×], Business[동의]
   → 기술팀의 우려: [구체적]
```

### STEP 2: 충돌 지점 명시화 (Conflict Mapping)

질문: "어디서 관점이 충돌하는가?"

아래 5가지 패턴을 확인:

**패턴 1: "사용자가 원하는" vs "비즈니스가 현실적인"**
- User가 강조하는데 Business가 미온적인 기능?
- 이유: [Business 관점 기술]
- 해결책: [절충안]

**패턴 2: "시장이 요구하는" vs "기술이 할 수 있는"**
- Market이 강조하는데 Tech가 시간/복잡도로 우려하는 기능?
- 이유: [Tech 관점 기술]
- 해결책: [MVP vs Phase 2 구분?]

**패턴 3: "시장 기회" vs "개발 타임라인"**
- Market의 시장 창 vs Tech의 개발 속도
- 우리가 이기거나 진다면?
- 대응: [구체적 액션]

**패턴 4: "사용자 성공" vs "비즈니스 지표"**
- User의 성공 정의가 Business의 KPI와 다른가?
- 예: "사용자 만족도"(User) vs "ARR 성장"(Business)
- 우선순위: [누가 먼저인가?]

**패턴 5: "완벽함" vs "빠름"**
- MVP로 출시할 때 "최소한의 품질"은 뭔가?
- 각 관점별 의견:
  - User: [최소 기능]
  - Tech: [최소 품질]
  - Business: [최소 고객 수]
  - Market: [최소 차별화]

### STEP 3: "우리는 뭘 빼야 하는가?" (Prioritization)

각 팀에게:
- "당신 관점에서 '절대 포함'되어야 할 Top 3?"
- "당신 관점에서 '미룰 수 있는' Top 3?"

**형식:**
```
MUST HAVE (이것 없으면 망함):
- Market: [1. __ 2. __ 3. __]
- User: [1. __ 2. __ 3. __]
- Tech: [1. __ 2. __ 3. __]
- Business: [1. __ 2. __ 3. __]

CAN WAIT (Phase 2로 미룰 수 있음):
- Market: [1. __ 2. __ 3. __]
- User: [1. __ 2. __ 3. __]
- Tech: [1. __ 2. __ 3. __]
- Business: [1. __ 2. __ 3. __]
```

### STEP 4: 위험 가정 포착 (Riskiest Assumptions)

질문: "우리 계획에서 가장 위험한 가정 5가지는?"

각 관점에서:
- Market: "이 가정이 틀리면 시장 기회 자체가 사라진다"
- User: "이 가정이 틀리면 사용자가 안 쓴다"
- Tech: "이 가정이 틀리면 기술적으로 불가능해진다"
- Business: "이 가정이 틀리면 사업성이 없어진다"

**형식:**
```
위험 가정 TOP 5:

1️⃣ [가정]: "만약 [X]가 아니라면?"
   영향도: [높음/중간/낮음]
   발생확률: [높음/중간/낮음]
   검증 방법: [어떻게 확인할 것인가?]
   대응책: [만약 틀렸을 때 할 일]
   리스크 소유자: [누가 감시할 것인가?]

2️⃣ ...
```

### STEP 5: 최종 합의 도출 (Final Convergence)

## 팁

### 조사 단계에서
- 각 Teammate가 "근거"를 명확히 하도록 강조
- 추상적 답변은 "구체적 예시를 3가지 주세요"로 재요청
- 충돌하는 의견이 나오면 좋음 (건강한 신호)

### 토론 단계에서
- Moderator가 중립적이어야 함
- 한 팀의 의견이 독주하면 "다른 관점은?"으로 균형 조절
- "동의합니다"는 약한 신호 → "왜 동의합니까?"로 깊이 있게

### 수렴 단계에서
- Green Zone은 **보수적으로** (모두 동의한 것만)
- Yellow Zone은 **명확한 기준**으로 (언제 포함될 것인가?)
- Red Zone도 **언제 재검토할 것인가** 명시
